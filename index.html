Overview of I/O Organization

Computers communicate with the outside world through Input/Output (I/O) devices. The key challenge is that I/O devices are much slower than the CPU, and each has different speeds and behavior. Three main mechanisms handle this:

02

The Bus

Definition: A bus is a shared communication link that uses one set of wires to connect multiple subsystems (Processor, Memory, I/O Devices).

Two major advantages: Versatility and Low Cost.

Major disadvantage: It can become a communication bottleneck, limiting maximum throughput when too many devices share it.

Bus Line Types

Address Lines

Carry the address of the memory location or I/O device being accessed.

Data Lines

Carry the actual data, commands, or addresses between source and destination.

Control Lines

Signal requests, acknowledgments, and indicate type of transaction on data lines.

Synchronous vs Asynchronous Bus

Synchronous

Includes a clock signal in control lines. All devices run at the same clock rate. Fast and simple to implement.

⚠ Limitations: All devices must run at same rate. Cannot be long (clock skew problem).

Asynchronous

Not clocked. Supports wide variety of devices, can be lengthy. Uses handshaking protocol to coordinate data transfer.

✓ No clock skew issues. Works with slow and fast devices.

Asynchronous Handshaking — 3 Control Lines

ReadReq — I/O device asserts this to request a memory read. Address placed on data lines simultaneously.

DataRdy — Asserted to signal that data word is now ready on data lines. Asserted by both memory and I/O device at different steps.

Ack — Used to acknowledge either ReadReq or DataRdy signal from the other party.

Handshaking Steps (Read Operation)

  1. Memory sees ReadReq; reads address from data bus; raises Ack to confirm.
  2. Ack is high → I/O releases ReadReq and data lines.
  3. Memory sees ReadReq is low → drops Ack. (Memory read now in progress.)
  4. Memory has data ready → places data on data lines; raises DataRdy.
  5. I/O device sees DataRdy → reads data from bus; raises Ack.
  6. Memory drops DataRdy; releases data lines.
  7. I/O device sees DataRdy go low → drops Ack. Transfer complete!

I/O Interface

The I/O Interface is the hardware circuit that connects an I/O device to the bus. The processor places a unique device address on the address lines; the device that recognizes the address responds.

Components of I/O Interface

Address Decoder — Identifies if the address on the bus belongs to this device.

Control Circuits — Handles read/write commands from the processor.

Data Registers — DATAIN (input buffer), DATAOUT (output buffer).

Status Registers — Contains flag bits like SIN, SOUT, KIRQ, DIRQ.

I/O Mechanisms — Big Picture

There are 4 I/O mechanisms, each progressively offloading work from the CPU:

1 Memory-Mapped I/O

I/O devices share the same address space as memory. Accessed using normal memory instructions.

2 Programmed I/O (Polling)

CPU directly controls I/O, constantly checking status (polling). Wastes CPU time.

3 Interrupt-Driven I/O

I/O device signals CPU when ready. CPU doesn't waste time waiting — does other work.

4 DMA (Direct Memory Access)

Special controller transfers data directly between device and memory without CPU intervention. Best for bulk transfers.

Memory-Mapped I/O

Key Idea: I/O devices and Memory share the same address space. Portions of the address space are assigned to I/O devices. Reading/writing to those addresses sends commands to the device.

Uses the same address bus always — no separate I/O instructions needed. Normal Move instructions work for I/O.

Address Space Layout

0x0000Main Memory (RAM)
...Main Memory continues...
DATAINKeyboard Input Buffer ← I/O Device
DATAOUTDisplay Output Buffer ← I/O Device
STATUSStatus Register (SIN, SOUT flags)
CONTROLControl Register (KEN, DEN bits)
0xFFFFTop of Address Space

Example Instructions

; Read keyboard character into register R0 Move DATAIN, R0 ; reads from keyboard buffer → R0 ; Send character from R0 to display Move R0, DATAOUT ; writes R0 → display output buffer

🔐 Security: User programs are prevented from accessing I/O device addresses directly. The OS uses address translation to protect these addresses — only kernel/supervisor mode can access them.

Programmed I/O (Polling)

Concept: The CPU has direct control over all I/O operations. It repeatedly checks (polls) the status of the I/O device, waiting for it to be ready. This is called busy-waiting or polling.

ISA Assembly Example — Keyboard to Display

; Read one line from keyboard, store in buffer, echo to display Move #LINE, R0 ; Initialize memory pointer WAITK: TestBit #0, STATUS ; Test SIN flag (bit 0) Branch=0 WAITK ; If SIN=0, keyboard not ready, wait Move DATAIN, R1 ; Read character from keyboard WAITD: TestBit #1, STATUS ; Test SOUT flag (bit 1) Branch=0 WAITD ; If SOUT=0, display not ready, wait Move R1, DATAOUT ; Send character to display Move R1, (R0)+ ; Store character, advance pointer Compare #$0D, R1 ; Check if Carriage Return Branch≠0 WAITK ; If not CR, get next character Move #$0A, DATAOUT ; Send Line Feed Call PROCESS ; Process the input line

⚠ Problem with Polling: The processor may poll the status register hundreds of thousands of times before the slow I/O device is ready. This wastes CPU time that could be used for useful computation.

07

Interrupt-Driven I/O

Key Idea: Instead of the CPU constantly checking the device, the device notifies the CPU when it needs attention by raising an interrupt signal. The CPU can do useful work in the meantime.

Interrupt Latency = time to save status + restore context before executing ISR (Interrupt Service Routine).

Interrupt Handling Sequence

  1. The I/O device raises an interrupt request signal (INTR).
  2. Processor finishes current instruction, then suspends current program.
  3. Interrupts disabled (to prevent nested interrupts during ISR setup).
  4. Device informed its request is recognized; it deactivates the interrupt signal.
  5. Action requested is performed by the Interrupt Service Routine (ISR).
  6. Interrupts re-enabled; interrupted program resumes.

Handling Multiple Devices — 4 Techniques

1. Polling

ISR checks IRQ bit of each device in order. First device with IRQ set gets serviced. Simple but slow — checks all devices even if most aren't requesting.

2. Vectored Interrupts

Device sends its identity (Interrupt Vector) to processor. CPU uses vector to jump directly to the correct ISR address. Faster — no polling needed.

3. Interrupt Nesting

High-priority interrupt can pre-empt the ISR of a low-priority interrupt. Devices organized in priority levels. Processor only interrupted by higher-priority device.

4. Daisy Chaining

INTR line is common; INTA (acknowledge) propagates serially through devices. Device closest to processor gets highest priority. Risk: starvation for low-priority devices.

Enabling/Disabling Interrupts — 3 Methods

1. Ignore Interrupt — Processor hardware ignores interrupt until first ISR instruction completes. Interrupt disable instruction used after first ISR instruction.

2. Disable Interrupts — Processor automatically disables interrupts at start of ISR. Interrupt-enable bit set to 0. Restored when ISR returns.

3. Edge-Triggered Line — Processor receives only one request regardless of how long line stays active. No separate interrupt disable needed.

08

Direct Memory Access (DMA)

Problem with Interrupts: Still too much CPU overhead for bulk data transfers (e.g., copying a file from disk). Every byte/word requires an interrupt and CPU involvement.

DMA Solution: A special DMA Controller takes over the bus and transfers data directly between the device and main memory, with minimal CPU involvement.

DMA Controller Registers

Starting Address Register — Memory address where block transfer begins.

Word Count Register — Number of words to transfer.

Status & Control Register — Mode (R/W), IRQ flag, IE (interrupt enable), Done flag.

DMA Transfer Process

  1. CPU sends DMA controller: starting address, word count, R/W mode, and start signal.
  2. DMA controller takes bus control ("steals" cycles from CPU).
  3. DMA transfers entire block of data directly between device ↔ memory.
  4. DMA controller sends an interrupt to CPU when transfer is complete.
  5. CPU resumes normal operation.

DMA Transfer Modes

Cycle Stealing

DMA "steals" one memory cycle at a time from the processor. Processor and DMA accesses are interwoven. DMA has higher bus priority.

Burst / Block Mode

DMA controller gets exclusive access to main memory for entire block transfer without interruption. Faster but processor locked out longer.

⚠ DMA Conflicts: (1) Processor vs DMA controller — both want bus simultaneously. (2) Two DMA controllers both trying to access memory at same time. Resolved by Bus Arbitration.

Bus Arbitration

Bus Master — The device that currently controls the bus and initiates data transfers.

Bus Arbitration — The process of selecting which device gets to become the next bus master when multiple devices want the bus simultaneously.

Key Bus Arbitration Signals

BR (Bus Request) — Open drain line. Any DMA device pulls this low to request bus. Logical OR of all device requests.

BG (Bus Grant) — Processor activates this to acknowledge bus request. Propagates in daisy chain to DMA controllers.

BBSY (Bus Busy) — Open collector line. Current bus master asserts this to show it's using the bus.

Centralized Arbitration

A separate unit (arbiter circuit) controls all bus access. Processor is normally the bus master and grants mastership to DMA. Simple and common.

Distributed Arbitration

No central arbiter. All devices participate. Each device has a unique ID (ARB0–ARB3). Device with highest ID wins arbitration by comparing bits on arbitration lines.

Distributed Arbitration Example

Device A (ID = 5 = 0101) and Device B (ID = 6 = 0110) both request bus:

  1. Both place their IDs on ARB lines: A→0101, B→0110. Combined pattern: 0111
  2. Device A compares 0111 to its ID 0101 — detects difference on ARB1 bit.
  3. Device A disables its drivers for ARB1 and ARB0.
  4. Pattern becomes 0110 — Device B's ID. Device B wins!

Universal Serial Bus (USB)

USB was designed to provide a simple, low-cost, easy-to-use interface that overcomes the limited number of I/O ports. It supports plug-and-play operation.

USB Speeds

VersionSpeed NameData RateUse Case
USB 1.0Low Speed1.5 Mbit/sHID: keyboard, mouse, joystick
USB 1.1Full Speed12 Mbit/sBasic USB data rate
USB 2.0High Speed480 Mbit/sDrives, cameras (introduced 2001)
USB 3.0SuperSpeed5 Gbit/sHigh-speed storage (~572 MB/s)

USB Tree Structure

Key USB Principles

🌳 Tree Structure — Root hub connects to host. Hubs act as intermediate controllers. Leaves are I/O devices. Up to 127 devices with 7-bit addresses.

📢 Broadcast — Host messages are broadcast to ALL devices via hubs. Only the addressed device responds. Device-to-device communication is NOT possible.

🔄 Polling-Based — USB operates strictly on polling. A device may only send a message in response to a poll from the host. No conflicts from simultaneous upstream messages.

⚡ Speed Matching — Messages on high-speed links are always transmitted at high speed, even for low-speed destination devices. Hub then forwards at low speed.

USB Address Assignment (Plug-and-Play)

  1. New device connected to hub — starts with address 0.
  2. Hub detects new device, records in its status information.
  3. Host periodically polls hubs to find new devices.
  4. Host sends commands to configure the device, reads its capabilities.
  5. Host assigns device a unique 7-bit USB address.
  6. Device responds only to its new address from this point on.
